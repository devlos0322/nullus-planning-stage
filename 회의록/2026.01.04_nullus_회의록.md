### 3. 데이터베이스 선택 (이기하 님 제안)

#### 3.1 PostgreSQL 제안
**장점**:
- 다양한 플러그인 지원
- 벡터 DB 플러그인 제공 (pgvector)
- SQL 기반 쿼리 (별도 벡터 DB 언어 학습 불필요)
- 오픈소스 (벤더 독립적)
- 성능: 벡터 DB 플러그인 성능도 양호

**비교**:
- MySQL/MariaDB: 플러그인 부족
- Oracle: 벤더 종속

#### 3.2 의견 교환
**이제응 님 의견**:
- PostgreSQL의 장단점을 객관적으로 비교 필요
- 벡터 DB 성능 비교 필요
- PostgreSQL이 벡터 DB 성능에서 약할 수 있음

**이기하 님 대응**:
- 가능한 옵션으로 제안
- 더 좋은 게 있으면 그걸로 변경 가능
- 비교 분석표 작성 필요
- 회사 시너지 고려 (EDB 등)

**결론**:
- PostgreSQL을 기본으로 하되, 비교 분석 후 최종 결정
- 벤터 DB 성능 비교 분석 필요
- 회사 시너지 고려

### 4. 데이터 옵스 & ML 옵스 논의

#### 4.1 데이터 옵스 필요성 (이제응 님 질문)

**질문 내용**:
- 데이터 옵스 레이어가 필요한지?
- 데이터 성향별 처리 모듈화 필요
- 특정 도메인이 아닌 데이터 성향 기반 설계

**데이터 성향 분류 제안**:
1. **작은 데이터 대량**: 이커머스 트래픽, 채팅 등
2. **큰 데이터**: 연구소 (10GB~100GB 파일)
3. **고빈도 트랜잭션**: 초당 많은 데이터
4. **저빈도 주기적**: 천천히 주기적으로 들어오는 데이터

**목표**:
- 데이터 소스가 없어도 설계만으로 모듈화
- 오픈소스 커뮤니티의 기여 유도
- 모듈러 구조로 확장 가능

#### 4.2 이기하 님 의견

**현실적 제약**:
- 데이터를 구하기 어려움
- 특정 기업/도메인 데이터 필요

**해결 방안**:
- **AI 기반 가상 데이터 생성**
  - 작은 데이터 대량 / 큰 데이터 / 정형/비정형
  - 이미지 등 비정형 데이터 결합
  - 실제 데이터 흐름 시연

**예시**:
- 공장 데이터 (삼성전자)
- 통신사 데이터 (테라바이트)
- 공공 데이터 (기상청, 자전거 데이터 등)

**결론**:
- 실제 구현된 코드 제공이 중요
- AI 기반 가상 데이터 생성 활용
- 여러 상황별 예시 제공

#### 4.3 박준희 님 경험 공유

**공공기관 AIOps 플랫폼 구축 경험**:
- **4개 레이어 구조**:
  1. **GPU Ops**: NVIDIA GPU Operator, Prometheus 모니터링
  2. **데이터 레이어**: 데이터셋 포맷/라벨링/클렌징/저장
  3. **ML Ops**: Apache Airflow 기반 파이프라인
     - 데이터 준비
     - 퍼블릭 모델 하이퍼파라미터 튜닝
     - 모델 평가
     - 모델 코드 저장
     - 주기적 훈련 및 배포
  4. **LLM Ops**: LangChain 기반
     - 학습된 모델 기반 프롬프트 품질 조정
     - 지속적 버전 관리

**한계**:
- LangChain이 GRPC로 API 제공
- 모든 기능을 API로 제공하지 않음
- 올해 필요한 기능만 추출하여 커스터마이징 계획

#### 4.4 허유정 님 의견

**제안**:
- 데이터 옵스, DevOps, ML Ops 레이어를 나눠서 정리 후 통합
- API, 포털 플러그인 형태로 서비스

**결론**:
- 각 레이어별로 정리 후 통합
- FinOps 추가 검토 필요
- API 중심 설계 강조

### 5. 개발 환경 및 배포 전략 (이기하 님)

#### 5.1 로컬 우선 전략

**배경**:
- 초기 자원 제약
- 로컬 환경 지원 필요
- 맥 사용자 지원 (ARM64 이미지 부족)

**구현 계획**:
1. **OS 이미지 최적화**:
   - Ubuntu 24.04 기반
   - Kubernetes 최적화 설정
   - 베이스 이미지 생성

2. **가상화 지원**:
   - VirtualBox (무료)
   - VMware Fusion (무료)
   - Packer를 이용한 이미지 빌드
   - HashiCorp Vagrant Cloud 배포 (무료)

3. **멀티 아키텍처 지원**:
   - ARM64 (맥)
   - AMD64/x86_64 (윈도우)
   - 각 아키텍처별 이미지 빌드

**예시**: Harbor
- 공식 ARM64 버전 없음
- Bitnami가 ARM64 버전 제공했으나 중단
- 소스를 이용한 ARM64 빌드 가능

#### 5.2 코드 기반 자동화

**목표**: Kubeflow 스타일
- **한 줄 명령어로 전체 설치**
- **90% 이상 자동 설치**
- 사용자 부담 최소화

**기술 스택**:
- Terraform/OpenTofu
- Ansible
- Shell Script

**컨셉**:
- 기본 세팅은 자동 설치
- 필요한 설정만 변경
- 즉시 사용 가능한 상태 제공

- **참고**: Chasm 프로젝트 매트릭스

### 전문가 네트워크
- **파코** (중국): CPO/CTO, 클라우드 아키텍처 전문가, QCon 만남
- **텔레미토리**: 공헌도 높은 전문가 (참고 인물)
- **추가 필요**: 글로벌 전문가 1명 (LinkedIn 등으로 추천 요청)

### 정부 및 기관 연계
- **과기부**: 정부 차원 지원 가능
- **국회**: 오픈소스 대회 주최 (위원으로 참여)
- **CNCF**: 글로벌 커뮤니티 확장
