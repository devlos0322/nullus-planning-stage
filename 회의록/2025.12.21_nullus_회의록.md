## 💬 주요 논의 사항

### 1. 프로젝트 방향성 (이기하 님 의견)

#### 핵심 아이디어
- **아이디어 수준이 아닌 실제 구현된 코드 제공**
- 인터넷에 떠도는 표준들을 실제 구현
- 예: Kubernetes 컴포넌트 조합을 실제 코드로 구현

#### 구체적 접근
- **오픈소스 조합**: Kubernetes + ArgoCD + Harbor 등
- **실체화된 코드 제공**: 
  - 70-80% 이상 완성도
  - "한 줄 명령어로 설치 완료" 수준
  - 기본 예제까지 포함하여 즉시 사용 가능

#### 예시
```
Kubernetes 설치 → ArgoCD → Harbor 등 조합
→ 실제 배포 가능한 완성된 코드 제공
→ 사용자가 바로 활용 가능한 형태
```

#### 기대 효과
- 표준처럼 흘러가는 것
- 실체화된 솔루션으로 인정받음
- 재사용성과 확장성

### 2. 우려사항 및 해결 방안 (이제응 님)

#### 우려사항

**2.1 컴포넌트 버전업 문제**
- 각 컴포넌트 버전업에 따른 연동 문제
- 예: KT 클라우드의 OKD 버전 상품화 실패 사례
- 예: 삼성 SDS의 SPCC 프로젝트 실패 (수천억 원 투자)

**2.2 오픈소스 생태계의 빠른 변화**
- Kong → Traefik, Istio 등 생태계 변화
- 커뮤니티 활동에 따른 기술력 변동
- 예: Kong → Traefik 전환, 크라캔디 (LG전자 사용 중)

**2.3 향후 5년 후 AI 발전**
- AI가 코드 생성 능력 향상
- 현재 만든 것들의 유지보수 필요성
- Vibe Coding 시대 대비

**2.4 프레임워크 선택의 어려움**
- Backstage 기반 vs 스크래치
- Crossplane vs OpenTofu 등 선택
- 어떤 것을 중심 프레임워크로 할지?

#### 해결 방안 (이기하 님)

**2.1 Loose Coupling 설계**
- CNCF 사례: 200개 이상 프로젝트, 시대에 맞게 변화
- Tight Coupling 피하기 (아콘 제품 사례)
- 유기적이고 유연한 구조

**2.2 표준 선정**
- API Gateway 표준화 사례 참고
- S3 표준화 사례 참고
- 변하지 않는 표준 선정 후 유기적 관리

**2.3 구체적 실천**
- 현재 AI는 기존 코드 분석이 더 뛰어남
- 기존 코드가 있으면 AI가 더 잘 활용
- 아이디어만으로는 부족, 실체화 필요

**2.4 소프트웨어는 유기물**
- 지속적인 관리 필요
- 변화에 유연하게 대처할 수 있는 설계

### 3. 참여 의견

#### 허유정 님 (참석자 4)
- **참여 계기**: 사내 DevOps 플랫폼 개발/운영 경험
- **목표**: "이게 안 돼서 안 쓴다"는 피드백을 받지 않는 플랫폼
- **궁금증**: Backstage처럼 기존 유명 프로젝트 확장 vs 스크래치

#### 박준희 님 (참석자 5)
- **참여 계기**: 배우기 위해 참여
- **경험**: 클라우드넷 스터디, Terraform/OpenTofu, IaC 이해
- **공감**: 표준화된 전체를 아우르는 플랫폼 필요
- **고민**: AI가 대체 가능한 부분, 관리의 중요성

### 4. 추가 제안사항 (이제응 님)

#### 4.1 모듈 마켓플레이스
- **문제**: 플랫폼 엔지니어링이 IDP용으로 많이 사용
- **제안**: 다양한 모듈들이 들어가는 마켓플레이스 구축
- **기대**: 다른 회사 모듈 재활용, 확장성 향상

#### 4.2 도메인 및 인프라

- **러스트 코스트 사례**: 1차 GA 버전 완성, 기업 배포 예정
